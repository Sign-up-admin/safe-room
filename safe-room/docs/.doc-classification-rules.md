## 📖 概述

### 目的

建立统一的文档分类体系，确保文档有序组织、便于查找和维护。

### 范围

适用于 `docs/` 目录下的所有 Markdown 文档，不包括模板文件和脚本文件。

### 分类原则

1. **功能导向**：按文档功能和使用场景分类
2. **用户导向**：考虑不同用户群体的需求
3. **维护导向**：便于文档的更新和维护

## 🗂️ 分类体系

### 1. 需求文档 (requirements/)

#### 定义
描述系统功能需求、用户界面设计、业务逻辑规范的文档。

#### 适用场景
- 功能规格说明
- 用户界面设计
- 业务流程描述
- 验收标准定义

#### 子分类
- `requirements/frontend/`：前端需求文档
- `requirements/admin/`：管理后台需求文档
- `requirements/common/`：通用需求文档

#### 示例文档
- `USER_CENTER_REQUIREMENTS.md`
- `ADMIN_LOGIN_REQUIREMENTS.md`
- `PAYMENT_REQUIREMENTS.md`

### 2. 技术文档 (technical/)

#### 定义
描述系统技术实现、架构设计、接口规范的文档。

#### 适用场景
- 系统架构设计
- API 接口规范
- 数据库设计
- 部署配置

#### 子分类
- `technical/architecture/`：架构文档
- `technical/api/`：API 文档
- `technical/database/`：数据库文档
- `technical/deployment/`：部署文档

#### 示例文档
- `ARCHITECTURE.md`
- `API.md`
- `DATABASE_SCHEMA.md`
- `DOCKER.md`

### 3. 开发文档 (development/)

#### 定义
指导开发过程、测试规范、项目管理的文档。

#### 适用场景
- 开发指南
- 测试策略
- 代码规范
- 项目管理

#### 子分类
- `development/guides/`：开发指南
- `development/testing/`：测试文档
- `development/contributing/`：贡献指南

#### 示例文档
- `DEVELOPMENT.md`
- `TESTING_STRATEGY.md`
- `CI_CD_GUIDE.md`

### 4. 报告文档 (reports/)

#### 定义
记录项目进展、测试结果、代码审查的文档。

#### 适用场景
- 测试报告
- 代码审查记录
- 性能分析报告
- 项目进度报告

#### 子分类
- `reports/test-reports/`：测试报告
- `reports/code-reviews/`：代码审查
- `reports/performance/`：性能报告

#### 示例文档
- `TEST_COVERAGE_REPORT.md`
- `CODE_REVIEW_SUMMARY.md`
- `PERFORMANCE_ANALYSIS.md`

## 🔍 分类判断规则

### 基于内容特征判断

#### 需求文档特征
- 包含功能描述、用户故事
- 描述"应该做什么"（What）
- 包含验收标准
- 面向业务人员和产品经理

#### 技术文档特征
- 包含技术实现细节
- 描述"如何做"（How）
- 包含代码示例、配置信息
- 面向开发人员和运维人员

#### 开发文档特征
- 包含开发流程、规范
- 描述"如何协作"（Process）
- 包含工具使用、环境配置
- 面向团队成员

#### 报告文档特征
- 包含数据分析、结果记录
- 描述"发生了什么"（What happened）
- 包含时间戳、度量指标
- 面向管理者和审计人员

### 基于文件位置判断

#### 自动分类规则
1. `docs/requirements/` → requirements
2. `docs/technical/` → technical
3. `docs/development/` → development
4. `docs/reports/` → reports
5. `docs/` 根目录 → 根据内容特征判断

#### 优先级
1. 显式目录位置（最高优先级）
2. 文件名特征匹配
3. 内容特征分析（最低优先级）

### 基于文件名判断

#### 需求文档命名模式
- `*REQUIREMENTS.md`
- `*_REQUIREMENTS.md`
- `ADMIN_*REQUIREMENTS.md`（管理后台）

#### 技术文档命名模式
- `*ARCHITECTURE.md`
- `API*`
- `DATABASE*`
- `*DEPLOYMENT.md`

#### 开发文档命名模式
- `*GUIDE.md`
- `*STRATEGY.md`
- `TESTING*`
- `CI*`

#### 报告文档命名模式
- `*REPORT*`
- `*REVIEW*`
- `*ANALYSIS.md`

## 📝 文件命名规范

### 通用命名规则

#### 大小写规范
- 使用大写字母开头的单词
- 多个单词用下划线 `_` 分隔
- 避免使用空格和特殊字符

#### 文件扩展名
- 所有文档使用 `.md` 扩展名
- 模板文件使用 `.md` 扩展名

#### 长度限制
- 文件名长度不超过 50 个字符
- 路径长度不超过 100 个字符

### 分类特定命名规范

#### 需求文档
```
[模块名称]_REQUIREMENTS.md
ADMIN_[模块名称]_REQUIREMENTS.md
```

**示例**：
- `USER_CENTER_REQUIREMENTS.md`
- `ADMIN_LOGIN_REQUIREMENTS.md`
- `PAYMENT_REQUIREMENTS.md`

#### 技术文档
```
[主题]_ARCHITECTURE.md
API.md / API_REFERENCE.md
DATABASE.md / DATABASE_SCHEMA.md
[组件]_DEPLOYMENT.md
```

**示例**：
- `SYSTEM_ARCHITECTURE.md`
- `API_REFERENCE.md`
- `DATABASE_SCHEMA.md`
- `DOCKER_DEPLOYMENT.md`

#### 开发文档
```
[主题]_GUIDE.md
TESTING_[主题].md
CI_CD_GUIDE.md
```

**示例**：
- `DEVELOPMENT_GUIDE.md`
- `TESTING_STRATEGY.md`
- `CI_CD_GUIDE.md`

#### 报告文档
```
[主题]_[类型]_REPORT.md
CODE_REVIEW_[项目].md
PERFORMANCE_ANALYSIS.md
```

**示例**：
- `TEST_COVERAGE_REPORT.md`
- `CODE_REVIEW_BACKEND.md`
- `PERFORMANCE_ANALYSIS.md`

## 📁 目录结构规范

### 标准目录结构

```
docs/
├── requirements/          # 需求文档
│   ├── frontend/          # 前端需求
│   ├── admin/             # 管理后台需求
│   └── common/            # 通用需求
├── technical/             # 技术文档
│   ├── architecture/      # 架构文档
│   ├── api/               # API 文档
│   ├── database/          # 数据库文档
│   └── deployment/        # 部署文档
├── development/           # 开发文档
│   ├── guides/            # 开发指南
│   └── testing/           # 测试文档
├── reports/               # 报告文档
│   ├── test-reports/      # 测试报告
│   └── code-reviews/      # 代码审查
├── templates/             # 文档模板
├── scripts/               # 自动化脚本
└── INDEX.md               # 主索引文件
```

### 目录命名规范

#### 英文命名
- 所有目录使用英文命名
- 单词用连字符 `-` 分隔
- 使用复数形式表示集合

#### 层次限制
- 目录层次不超过 3 级
- 避免过深的目录嵌套

### 特殊目录

#### 根目录文档
- `README.md`：项目说明
- `CHANGELOG.md`：变更日志
- `ARCHITECTURE.md`：系统架构总览
- `CONTRIBUTING.md`：贡献指南

#### 保留目录
- `.doc-*`：文档工程配置文件
- `templates/`：文档模板
- `scripts/`：自动化脚本

## 🔄 分类变更流程

### 变更触发条件

#### 自动变更
- 文档移动到新目录时自动变更分类
- 文档内容特征发生显著变化时

#### 手动变更
- 文档功能发生变化
- 分类规则更新导致的重新分类
- 文档重构导致的分类调整

### 变更流程

#### 1. 评估变更影响
- 检查文档的关联关系
- 评估对其他文档的影响
- 确认变更的合理性

#### 2. 执行变更
- 更新文档头部分类字段
- 移动文档到对应目录（如果需要）
- 更新文档中的相对链接

#### 3. 更新关联文档
- 更新引用该文档的链接
- 更新相关索引文件
- 更新文档关系映射

#### 4. 验证变更
- 运行文档校验工具
- 检查链接有效性
- 确认文档可正常访问

### 变更记录

#### 变更日志格式
```
日期: YYYY-MM-DD
文档: [文档路径]
原分类: [原分类]
新分类: [新分类]
变更原因: [原因描述]
影响范围: [影响的文档列表]
```

#### 示例
```
日期: 2025-11-16
文档: docs/USER_MANUAL.md
原分类: technical
新分类: development
变更原因: 文档内容主要面向开发者使用
影响范围: docs/INDEX.md, docs/development/INDEX.md
```
